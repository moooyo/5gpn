# CLAUDE.md

Project guidance for working in this repo (5gpn — 5gpn-dns DoT/DoH/plain-53 gateway with mihomo-managed multi-exit egress).

## TUI / installer interaction: use Gum

All interactive UI and styled output in shell scripts (chiefly `install.sh`) is drawn with **[Gum](https://github.com/charmbracelet/gum)** (charmbracelet/gum). When adding or changing any prompt, menu, confirmation, spinner, or status output, use gum — do **not** add raw `read`/`echo`/`whiptail`/`dialog` as the primary path.

**Every operator-facing script follows the gum-or-echo pattern**, not just `install.sh`:

- `install.sh` carries the canonical inline helpers (`info/ok/warn/err/ask_*/gum_spin/card`) plus `install_gum()`.
- The sub-scripts it invokes (`scripts/update-lists.sh`, `setup-firewall.sh`, `gen-ios-profile.sh`, `renew-hook.sh`) each carry a **small self-contained gum-or-echo preamble** — they only *detect* gum (`command -v gum` + `[ -t 1 ]`), they never install it (that is `install.sh`'s job). Kept self-contained on purpose: no shared-lib sourcing failure mode, and the per-script duplication is locked against drift by `tests/test_tui_policy.sh`.
- `quick-install.sh` runs **before** gum is bootstrapped, so it is gum-aware-if-present with an ANSI fallback — do not make it install gum.
- **No Python left to exempt** (2026-07-01): the former terminal-TUI exemptions — `src/ios-http.py` and `tgbot.py` — are **gone**. Both are now in-process Go components of the `5gpn-dns` daemon (the iOS `.mobileconfig` responder on :8111; the Telegram bot via `github.com/go-telegram/bot`, a Telegram-native inline-keyboard TUI that needs no terminal gum). There is no Python in this repo.

Follow the established pattern (see `install.sh`):

- **Bootstrap, don't assume.** `install_gum()` downloads a **prebuilt** gum binary (no Go toolchain, no apt repo) and verifies it (`GUM_SHA256` override else the release `checksums.txt`); it runs early. `GUM_VERSION` is env-overridable. It must be **non-fatal under `set -euo pipefail`** — any failure leaves `_HAVE_GUM=0` and returns 0.
- **Always provide an echo fallback.** The `info/warn/ok/err` helpers branch on `_HAVE_GUM`: gum when `1`, plain ANSI `echo` when `0`. Never let output depend on gum being present.
- **Gate every gum *interaction* on a TTY: `[[ -t 0 ]]`.** `gum input/choose/confirm` need a terminal. **`main()` runs `attach_tty` first**, which reattaches stdin to `/dev/tty` (`exec 0</dev/tty`) when the installer is *piped* (`curl | sudo bash`) but a controlling terminal exists — so a piped install still prompts for DOMAIN / GATEWAY_IP / XRAY_RESOLVER (this is *why* the `[[ -t 0 ]]` gates fire under `curl | sudo bash`; before it they were all skipped and the run aborted on the missing domain). Truly headless runs (no `/dev/tty`: CI, cloud-init, systemd) fall through to the env-var / non-interactive path and never block on input. Locked by `tests/test_install_policy.sh`.
- **Guard prompt captures against cancel.** `gum input`/`gum confirm` exit non-zero on Esc/Ctrl-C, so `var="$(gum_helper …)"` under `set -e` would abort the whole run. Always write `var="$(ask_text '…' || true)"` (empty-on-cancel), matching the existing `ask_text`/`ask_secret`/`ask_yesno`/`gum_spin` helpers.
- **`gum_spin` only wraps opaque waits** (e.g. binary downloads), never output-bearing commands whose stdout the operator needs (e.g. `update-lists.sh`).

The subscription fetcher's trust-host resolver delegates to the CURRENT trust group (`exchangerFunc` in main), so swaps propagate.
  - **Upstream groups query SEQUENTIALLY in pool order** (deliberate reversal, 2026-07-10, user-directed — replaces the fan-out-race `group.Exchange`): members are tried one at a time in CONFIGURATION order and the first success returns immediately — no concurrent fan-out, no latency-based winner. Why: the adopted answer must be deterministic (pool order = the operator's preference order), gateway-side latency says nothing about the CLIENTS' network position (clients are on the terminal side, not on the box), and racing burned N-1 wasted queries per lookup. Each attempt gets an equal slice of the remaining ctx budget (`remaining / members-left`) so a dead/silent first member cannot starve the later ones; unused budget rolls over. The H1 rule is preserved (parent-ctx `Canceled` → abandoned, never a breaker failure; attempt-slice `DeadlineExceeded` on the CHILD keeps iterating); ECS attach/strip and 0x20 randomize-once semantics unchanged. Locked by `TestGroupExchangeSequentialPoolOrder` + `TestGroupExchangeFallsBackPastDeadMember` (real sockets). Cross-GROUP china∥trust concurrency in `Arbitrate` is UNCHANGED — only intra-group member selection became sequential.
  - **Resolve test** (`GET /api/resolve-test?domain=`, console 查询 view renamed **解析测试**): probes EVERY configured upstream server individually (`resolvetest.go`; china + trust, per-server proto/IPs/rcode/latency/error), adopts each group's FIRST-in-pool-order success (`selectFirst` — matching the sequential live behavior above; NEVER by measured latency, which is shown as a gateway-side diagnostic column only), re-applies the arbitration rule (china wins iff its answer has a CN address) and reports `chosen`/`chosen_ips`/`client_ips` (post-gateway-rewrite view). Terminal name verdicts (block/blacklist) probe nothing, matching the real pipeline. `/api/lookup` is kept (bot parity) but the SPA no longer uses it.
  - **Query log** (`querylog.go` + `GET /api/querylog?q=&limit=`, new console view **日志检索**): every answered query is recorded in an in-memory ring (cap 8192) with a **fixed 5-minute retention** — deliberately ephemeral, nothing on disk. Entries carry client IP, qname/qtype, verdict/reason (`resolveInfo` threaded through `resolveTraced`; `arbitrateSrc` exposes which group's answer was adopted), cache-hit flag, rcode, answer IPs, latency. The trace wrapper keeps the old `resolve(ctx,q,r)` signature for tests.

The systemd unit has `ReadWritePaths=/etc/5gpn` (whole conf dir, with `dns.env`/`cert`/`/opt/5gpn/web` read-only carve-outs) so the subscription manager can atomically create+rename caches AND `subscriptions.json`. The earlier "manual-only, don't implement subscriptions until the Phase 2 spec is written" stance is **reversed** — subscriptions are implemented. **Match types** (later addition, same phase family): manual rules for the three domain categories (`block`/`direct`/`blacklist`) now support four match types — `suffix` (default, bare `<cat>.txt`, label-bounded parent-domain/self match), `exact` (`<cat>.exact.txt`, whole-name equality), `keyword` (`<cat>.keyword.txt`, substring anywhere in the FQDN), `prefix` (`<cat>.prefix.txt`, raw prefix match, no label alignment) — via explicit per-file-name specs (`cmd/5gpn-dns/rules.go` `MatchType`/`DomainSet`, `main.go` `globPattern`), never a `<cat>.*.txt` wildcard. Subscription caches (`<cat>/*.txt`) stay `suffix`-only — remote lists carry no match-type info. `chnroute` is CIDR-only and has no match types. Surfaced everywhere rules are managed: `GET /api/rules/{cat}` returns a type-grouped object, `POST`/`DELETE /api/rules/{cat}` take a `type` field/query param (default `suffix`), and both the Telegram bot and the React web console expose all four types.

- **Subscriptions — resolver + retry fix**: the subscription fetcher resolves each subscription URL's hostname via the **trust DoT upstream** and dials the returned real IP directly (`HostResolver`/`trustHostResolver` in `cmd/5gpn-dns/subscription.go`), instead of the box's own system resolver. This fixes a self-reference deadlock: on a running gateway, the default resolve path rewrites a foreign subscription host to `DNS_GATEWAY_IP`, which the SSRF dial guard then correctly refuses as an internal address — subscriptions could never fetch on a fully-configured box. Trust-resolve failure falls back to the system resolver (offline-tolerant, e.g. tests with no trust upstream); the final dial target is still SSRF-checked. Failed fetches now retry with **1m → 30m exponential backoff** (`nextBackoff`, doubles, caps at 30m) instead of waiting for the subscription's full `interval`, so a transient outage recovers quickly.

- **Encrypted-DNS / HTTPDNS bypass block** (2026-07-04, default-on): `install.sh` seeds a `block` bypass list (`etc/block-dns-bypass.txt` → `block.txt` suffix + `etc/block-dns-bypass.keyword.txt` → `block.keyword.txt` `httpdns`/`httpsdns` keywords) so a client that uses its **own DoH/DoT or app HTTPDNS** gets NXDOMAIN and falls back to 5gpn-dns. Contents: common browser DoH/DoT providers (Cloudflare/Google/Quad9/AdGuard/NextDNS/OpenDNS/Mullvad/CN doh.pub-alidns-360) + Firefox canary `use-application-dns.net` + the **domain** rules of kelee.one Block_HTTPDNS. **Fundamental limit (must not paper over):** this is NAME-based only — a resolver reached by **hardcoded IP** (many app HTTPDNS; DoH bootstrap IPs) is NOT blockable, because in the peer topology (`setup-firewall.sh` `forward` chain `policy drop`, gateway is not the clients' router) client→third-party traffic never transits the gateway, so nftables can't drop it. **And this blind spot is INHERENT for 5gpn's actual client — an unmanaged iOS phone on cellular:** iOS-cellular has NO default-gateway/route knob, so forcing all client traffic through the box would need a VPN (forbidden "exit layer") or the 5G NPN operator routing clients through it at the core/UPF (operator-side infra, not a phone or gateway setting). ## Linux test environment (test-env)

A real Debian box for the Linux/CI gates that can't run on the Windows dev box (`mihomo -t` / `nft -c` / live DNS + cert/renewal flows, full `install.sh`). Its host/IP/credentials are kept out of this repo — configure a `test-env` SSH host alias locally (private config) and refer to it by alias only.

- **Host:** a Debian 13 (trixie) x86_64 box reached via the `test-env` SSH alias. The concrete address and auth are private — do not commit them.
- **SSH login:** `ssh test-env` (alias resolved from your own private SSH config + agent).
- **⚠️ Must use Windows native `ssh.exe`**, invoked via PowerShell:
  `& "$env:WINDIR\System32\OpenSSH\ssh.exe" test-env '<cmd>'`.
  **Do not use Git Bash's `ssh`** — it fails two ways: (1) a UTF-8 BOM at the start of `~/.ssh/config` makes it error `Bad configuration option: \357\273\277include`, and (2) it can't reach an agent private key held in a Windows named pipe.
- **Remote command quoting:** avoid `()` inside the PowerShell single-quoted command (it reaches remote bash as syntax). For multi-line / complex remote commands, the most robust path is to **base64-encode the script locally and decode remotely**: `$b64=[Convert]::ToBase64String([Text.Encoding]::UTF8.GetBytes($script)); & ssh.exe test-env "echo $b64 | base64 -d | bash"`. This dodges two Windows-PowerShell-5.1 traps that a here-string `@'…'@` piped to `bash -s` hits: a UTF-8 BOM prepended to the first line (mangles `set`/the first var), and the command-line length limit on large payloads. For whole files (configs, units, the built `5gpn-dns` binary), use `scp.exe` to copy them over rather than inlining.
- ⚠️ **Real box, not a sandbox** — treat it like production: don't run destructive commands without reason, and remember `install.sh` changes firewall/systemd/certs there for real.
